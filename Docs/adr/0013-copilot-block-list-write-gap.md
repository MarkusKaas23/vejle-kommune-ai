# Copilot read/write boundary for nested block lists and global elements

Tested on 2026-06-04 against Umbraco.AI.Agent.Copilot 1.0.0 with Gemini 2.5 Flash via OpenAI-compat routing. Two content structures were created in the VejleKommune demo: (A) an `accordionBlock` element type containing a nested `accordionItem` block list, added inline to a Content Page; and (B) an `accordionGlobalElement` standalone content node (global element pattern) containing the same nested block list, referenced from a Content Page via a Multi-Node Tree Picker.

**Test A — Copilot opened directly on the global element node.** Copilot correctly read all nested accordion item titles and body text through the block list hierarchy. When asked to rewrite a title, `set_value` was called twice and failed both times. The agent reported a technical error and instructed the editor to update manually.

**Test B — Copilot opened on the Content Page that references the global element via a picker.** Copilot recognised the picker value as a content reference, called `get_umbraco_content` automatically, and returned all three accordion item titles. When asked to edit a title it correctly identified the content as shared/reused and warned that changes would affect every page referencing it — then reported it could not edit from this context and offered to replace the shared element with a local copy instead.

**Conclusions.** The read path and cross-reference traversal are fully functional out of the box. The agent also demonstrated emergent understanding of the shared content concept without any custom instructions. The single confirmed gap is block-list-item-level writes: `set_value` operates on a property as a whole and cannot target a specific item at index N within a block list's JSON. This affects both direct edits (Test A) and edits across a reference boundary (Test B).

**Decision.** The Limbo.Umbraco.AI custom tool layer must implement exactly one tool to close this gap: `SetBlockListItemValue`. The tool takes a content node key, a property alias, a block item index, a field alias within the block's element type, and a new value. It reads the current block list JSON, mutates the target item, and writes the full JSON back via the management API. An optional `isSharedContent` flag (auto-detected by checking whether the target node is different from the context node) should prompt the agent to surface a confirmation to the editor before writing.

Everything else — reading flat properties, reading nested blocks, traversing picker references, listing referenced content — requires no custom tooling and is covered by the out-of-the-box agent.

**Amendment (2026-06-04, post-implementation).** The backend tool `SetBlockListItemValueTool` was implemented and registered via `builder.AITools().Add<>()`. It compiles and registers correctly. However, log inspection of the Copilot's `stream-agui` endpoint reveals that the Copilot's `auto` agent operates with a **hardcoded frontend tool set** (`set_value, add_item, remove_item, move_item, clear_value, save, save_and_publish, get_page_info, show_weather`) that is not extensible via `builder.AITools()`. Custom tools registered through that API are available to named agents (created in AI → Agents), but not to the built-in Copilot `auto` agent. The LLM therefore never receives `set_block_list_item_value` as a callable function and falls back to `set_value`.

**Revised decision.** Closing the write gap in the Copilot requires one of two approaches: (A) a TypeScript/Lit frontend tool handler registered as a custom Umbraco backoffice extension that intercepts the tool call and delegates to a backend HTTP endpoint — same architecture as the existing built-in tools; or (B) configuring the Copilot to use a named agent (if `Umbraco.AI.Agent.Copilot` supports agent aliasing in a future version). Path A is the production-grade route and is the planned approach for the Limbo.Umbraco.AI package. Path B should be monitored as Umbraco.AI evolves. The backend `SetBlockListItemValueTool` implementation is retained as the server-side half of Path A.

**Amendment 2 (2026-06-04, extended testing).** Additional tests established that the write gap is narrower than originally described. `set_value` was tested against three structures:

1. **Block Grid** (Hero Block with flat `heading`, `subheading`, `body`) — `set_value` **succeeded** on all three properties. Block Grid has native path-based property addressing in the Copilot frontend tools.
2. **Block List with flat element types** (TextBlock: `title`, `body`, `callToAction`, no nested blocks) — `set_value` **succeeded**. Simple single-level Block List items are writable out of the box.
3. **Block List with nested Block List** (AccordionItem containing its own `accordionItems` Block List) — `set_value` **failed**. The `set_value` path selector cannot navigate more than one level of block nesting.

**Refined conclusion.** The write gap applies only to Block List items that themselves contain another Block List property (multi-level nesting). Single-level Block List items and all Block Grid items are writable. For Limbo clients, this means:
- Standard module Block Lists (text, image, hero blocks) → no custom tooling needed
- Accordion / tab / FAQ patterns where items contain nested block lists → `SetBlockListItemValueTool` + frontend handler needed

**Human-in-the-loop observation.** When the Copilot calls `save` after writing, the backoffice renders an **Approve / Deny** button before the save is committed. This is the built-in human control gate in `Umbraco.AI.Agent.Copilot` 1.0.0. Individual `set_value` writes happen immediately without a gate, but the final `save` (and by extension `save_and_publish`) requires explicit editor approval. This is an important finding for governance: Limbo does not need to implement a separate approval layer for the final save step — it is already built in.
