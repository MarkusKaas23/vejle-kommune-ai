# Doctype bootstrap via IComponent; uSync exports the result

ADR-0010 commits to uSync for ongoing doctype management. That ADR is silent on initial bootstrap. We bootstrap the doctype taxonomy programmatically via an Umbraco notification handler that runs on `UmbracoApplicationStartedNotification`, creates the data types / compositions / doctypes / media types from a single C# source of truth if they don't already exist, and lets uSync export the result to `/uSync` on the same boot. The exported XML is committed to git; subsequent edits flow through uSync as ADR-0010 prescribes.

Considered: (a) hand-written uSync XML — the literal reading of ADR-0010. Rejected because hand-writing ~13 cross-referenced XML files for Umbraco 17 without runtime verification is error-prone and silent failures at import are easy to introduce; (b) doctype creation in the Umbraco backoffice followed by uSync export — works, but isn't reproducible across fresh environments without committing the initial XML, and the user wanted automation. The IComponent approach is more code than (b) but produces a deterministic, type-checked, PR-reviewable bootstrap that fresh clones can run.

This does NOT contradict ADR-0010: ADR-0010 is about ongoing management (editor backoffice edits flow through uSync without custom code). The bootstrap component is one-shot — it short-circuits when the doctypes already exist, so backoffice/uSync changes are not overwritten on subsequent boots.

Trade-off acknowledged: the C# definitions become the canonical source of the initial taxonomy. If a developer hand-edits the uSync XML and the doctype doesn't yet exist in the DB, the bootstrap will create from C# and the XML edit will be ignored. Resolution: do not edit the uSync XML directly until the first boot is complete and the doctypes exist; from that point on uSync is the editing surface.

**Scope deviation from ADR-0006 in this bootstrap (deferred to Phase 5/6):** the two Block List Editor data types prescribed by ADR-0006 — `AccessibilityMetadata.findings` and `ServicePage.selfServiceLinks` — are stubbed as plain text properties in the Phase 0+1 bootstrap. Neither is load-bearing for Phase 0+1's milestone (valid Schema.org JSON-LD on every page); both have natural homes in their respective Pattern phases (Async analyze for findings, ServicePage UX for self-service links). The `SelfServiceLink` and `AccessibilityFinding` element types are deferred too. When implemented, they will be added as a normal backoffice change that uSync exports.
