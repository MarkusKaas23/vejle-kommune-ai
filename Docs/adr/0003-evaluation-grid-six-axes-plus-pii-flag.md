# Evaluation grid: six axes plus a PII risk flag

The thesis evaluates each Pattern against a fixed grid: Latency, Editor control, Reversibility, Failure mode visibility, Cost, and Implementation complexity. PII risk is carried as a per-Pattern flag (none / possible / likely + mitigation note) rather than a 7th axis, and gets a dedicated paragraph in each pattern chapter because the municipality context makes it too important to bury in a table column. Three axes — Editor control, Failure mode visibility, Cost — are the headline axes that lead the thesis discussion; the other three appear in the comparison table but get less prose.

We rejected a 4-axis grid (Latency, Editor control, Governance, Implementation complexity) because "Governance" was overloaded — it conflated audit trail, accountability, rollback story, prompt-as-policy, and PII handling. Decomposing it into Reversibility + Failure mode visibility + the PII flag, and noting prompt-as-policy qualitatively for the Gate / validator Pattern, makes each claim falsifiable and gives examiners specific things to interrogate. The cost: a denser comparison table; mitigated by leading discussion with the three headline axes.
