# Multilingual content uses built-in culture variants; AI translations are unpublished by default

The Demo vehicle stores DA and EN content as Umbraco built-in culture variants of the same node — not as separate node trees and not as duplicated property sets. This is the Umbraco-blessed multilingual approach, matches what real kommune.dk sites use, and operationalises **Editor control** for the **Async transform** Pattern: the **Variant publish gate** (editor publishing the **Translated variant**) IS the editor-control mechanism. Reversibility is bounded — unpublishing/deleting the EN variant leaves the DA **Source variant** untouched. Per-doctype opt-in via the "vary by culture" flag means internal/admin doctypes (e.g., **SiteSettings**) stay DA-only.

**Translated variants** are unpublished by default. This bundles two purposes deliberately: (1) it's the Editor control gate for the Async transform Pattern, and (2) it's a citizen-safety guard against unreviewed AI content reaching a public municipal site. Both matter; bundling them is a feature, not a smell. An autopublish-on-translate flag exists as a configuration setting, not a structural change — the Pattern chapter shows a two-row sub-table demonstrating that the same Pattern instantiated with different governance settings scores differently on the Editor control axis (review-before = 4 vs. none = 1). That comparison is itself a thesis contribution.

Schema.org JSON-LD output reads the current culture from the variant context (`IPublishedContent`) and renders one block per page with `inLanguage` set accordingly. Multi-language JSON-LD blocks on the same page are not produced — that's a niche pattern not expected by validators.
