# Thesis framed as a pattern catalogue, not a feature list

The eight planned AI integrations were initially listed as features. We instead frame the thesis as a catalogue of architectural Patterns for AI-in-CMS, with each integration as a Use case exemplifying a Pattern. Evaluation is per-Pattern, not per-Use case; the Vejle Kommune site is the Demo vehicle that makes the Patterns concrete.

We considered a comparative per-feature study (risks becoming a vendor benchmark) and an integrated workflow demo (risks the demo masking which integration delivered the value). The pattern-catalogue framing is harder to reverse than it looks: it shapes codebase organisation (folders by Pattern, not by Use case), the evaluation harness (per-Pattern metrics), and the chapter structure of the thesis. The Cut rule — under time pressure, drop duplicate-Pattern Use cases before dropping Patterns — falls out of this framing.
